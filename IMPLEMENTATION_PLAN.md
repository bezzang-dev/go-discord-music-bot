# Go 기반 Discord 유튜브 음악 봇 구현 계획

## 문서 목적
이 문서는 개인 또는 소규모 서버에서 사용하는 Discord 음악 봇을 Go로 구현하기 위한 기준 문서다.
주요 목표는 다음과 같다.

- Discord 슬래시 커맨드 기반 음악 봇 구현
- 유튜브 URL 재생 및 검색어 기반 재생 지원
- 길드별 독립 큐 관리
- 로컬 환경에서 안정적으로 실행 가능한 구조 확보

이 문서는 구현 계획서이므로, 실제 코드는 포함하지 않는다.

## 범위
### 포함
- Go 기반 Discord 봇
- 슬래시 커맨드만 지원
- 유튜브 URL 재생
- 유튜브 검색어 재생
- 곡 큐, 스킵, 정지, 일시정지, 재개
- 현재 곡 및 큐 조회
- 길드 단위 플레이어 관리
- `yt-dlp`와 `ffmpeg`를 사용하는 오디오 파이프라인

### 제외
- Spotify 직접 재생
- 플레이리스트 전체 가져오기
- 데이터베이스 기반 영속 큐
- 웹 대시보드
- 대규모 공개 배포용 샤딩 구조

## 핵심 기술 선택
- 언어: Go
- Discord 라이브러리: `github.com/bwmarrin/discordgo`
- 유튜브 검색/메타데이터 추출: `yt-dlp`
- 오디오 변환: `ffmpeg`
- 실행 환경: 개인 PC 또는 로컬 서버
- 상태 저장: 메모리 기반

## 왜 Go로 가능한가
Go만으로 충분히 구현 가능하다. 다만 유튜브 스트림 추출과 오디오 변환까지 순수 Go로 직접 처리하는 것은 복잡성과 유지보수 부담이 크다. 따라서 실무적으로는 다음 조합이 가장 안정적이다.

- Go: Discord 이벤트 처리, 큐 관리, 동시성 제어
- `yt-dlp`: 유튜브 검색 및 스트림 URL 확보
- `ffmpeg`: Discord 송출용 오디오 변환

## 사용자 기능
### 슬래시 커맨드
- `/join`
  - 사용자가 접속한 음성 채널에 봇 입장
- `/play query:<string>`
  - 유튜브 URL 또는 검색어를 받아 곡을 큐에 추가
- `/skip`
  - 현재 곡 스킵
- `/stop`
  - 현재 재생 중단 및 큐 전체 삭제
- `/pause`
  - 재생 일시정지
- `/resume`
  - 일시정지 해제
- `/queue`
  - 현재 곡과 대기열 표시
- `/nowplaying`
  - 현재 재생 중인 곡 표시
- `/leave`
  - 음성 채널 퇴장

## 설정 값
환경 변수는 아래 기준으로 사용한다.

- `DISCORD_TOKEN`
- `GUILD_ID`
- `YTDLP_PATH`
- `FFMPEG_PATH`
- `LOG_LEVEL`

개발 초기에는 `GUILD_ID`를 사용해 특정 서버에만 슬래시 커맨드를 등록하는 방식을 기본값으로 한다.

## 시스템 구조
### 1. Discord 계층
- 슬래시 커맨드 등록
- interaction 수신 및 라우팅
- 사용자 응답 메시지 생성

### 2. 플레이어 계층
- 길드별 플레이어 생성 및 관리
- 현재 곡 상태 유지
- 큐 추가, 스킵, 정지, 일시정지, 재개 처리
- 재생 종료 시 다음 곡 자동 실행

### 3. 미디어 계층
- 입력값이 URL인지 검색어인지 판별
- `yt-dlp`를 사용해 메타데이터와 스트림 입력 URL 확보
- `ffmpeg` 프로세스 실행 및 종료 제어

### 4. 보이스 계층
- Discord 음성 채널 연결
- 오디오 프레임 송출
- 연결 종료 및 리소스 정리

## 권장 패키지 구조
```text
cmd/bot/main.go
internal/config
internal/discord
internal/player
internal/media
internal/voice
internal/logging
```

## 주요 타입 설계
### `Track`
- 제목
- 원본 URL
- 웹페이지 URL
- 재생 시간
- 요청자 정보
- 썸네일 URL

### `PlayerState`
- `idle`
- `playing`
- `paused`

### `Player`
- `Enqueue`
- `Skip`
- `Stop`
- `Pause`
- `Resume`
- `Queue`
- `NowPlaying`

### `Resolver`
- 검색어 또는 URL을 받아 `Track` 메타데이터와 스트림 입력 URL 반환

### `GuildManager`
- 길드 ID 기준 플레이어 생성 및 조회

## 동시성 설계
- 길드마다 하나의 플레이어 루프를 둔다.
- 각 플레이어는 명령 채널과 큐를 가진다.
- 재생 중인 작업은 `context.Context`로 취소 가능해야 한다.
- `skip`, `stop`, `leave` 호출 시 현재 재생 중인 외부 프로세스가 즉시 정리되어야 한다.

이 구조를 사용하면 길드별 재생 흐름이 서로 섞이지 않고, 고루틴과 외부 프로세스 생명주기를 명확하게 관리할 수 있다.

## 구현 단계
### 1단계: 프로젝트 초기화
- `go mod init`
- 기본 디렉터리 구조 생성
- 환경 변수 로더 작성
- Discord 세션 시작 및 종료 처리 추가

완료 기준:
- 토큰이 없으면 명확히 실패한다.
- 봇이 Discord에 로그인할 수 있다.

### 2단계: 슬래시 커맨드 골격
- 커맨드 등록
- interaction 핸들러 분리
- 공통 응답 함수 설계

완료 기준:
- `/join`, `/play`, `/skip`, `/stop`, `/pause`, `/resume`, `/queue`, `/nowplaying`, `/leave`가 등록된다.
- 각 커맨드는 최소한 정상적으로 응답한다.

### 3단계: 길드별 큐와 플레이어
- 길드별 플레이어 생성
- 메모리 큐 구현
- 현재 곡 상태 관리
- `skip`, `stop`, `pause`, `resume` 구현

완료 기준:
- 같은 길드에서 여러 곡을 순서대로 큐에 넣을 수 있다.
- 다른 길드의 큐와 상태가 분리된다.

### 4단계: 유튜브 검색과 URL 해석
- URL과 검색어를 구분
- `yt-dlp`로 메타데이터 추출
- 검색어는 `ytsearch1:` 방식으로 처리

완료 기준:
- 검색어로 첫 번째 결과를 가져올 수 있다.
- URL 입력 시 곡 정보를 읽을 수 있다.

### 5단계: 오디오 파이프라인
- `ffmpeg` 프로세스 실행
- Discord 송출 형식으로 오디오 변환
- 재생 종료 시 다음 곡 자동 재생

완료 기준:
- 실제 음성 채널에서 곡이 재생된다.
- `skip` 또는 `stop` 시 현재 프로세스가 정상 종료된다.

### 6단계: 사용자 경험 보강
- `/queue` 출력 정리
- `/nowplaying` 출력 정리
- 오류 메시지 구체화
- 로그 개선

완료 기준:
- 사용자가 실패 원인을 이해할 수 있다.
- 운영 중 문제 원인을 로그로 추적할 수 있다.

### 7단계: 실행 문서 정리
- Discord Developer Portal 설정 가이드
- 환경 변수 예시
- `yt-dlp`, `ffmpeg` 설치 가이드
- 로컬 실행 방법 작성

완료 기준:
- 새 환경에서 빠르게 실행 가능하다.

## 테스트 시나리오
### 단위 테스트
- URL/검색어 판별
- `yt-dlp` JSON 파싱
- 큐 FIFO 순서
- 상태 전이
- 비정상 명령 호출 시 오류 반환

### 통합 테스트
- 두 곡 enqueue 후 자동 다음 곡 재생
- `skip` 시 다음 곡으로 전환
- `stop` 시 큐 삭제와 재생 중단
- 길드별 상태 분리 확인

### 수동 테스트
- 음성 채널 미참여 상태에서 `/join`, `/play`
- URL 재생
- 검색어 재생
- 재생 중 큐 추가
- `pause`, `resume`
- `skip`
- `stop`
- `leave`

## 위험요소와 대응
### 유튜브 응답 형식 변경
- 대응: `yt-dlp` stderr 로그를 남기고, 미디어 계층을 분리한다.

### 외부 바이너리 설치 누락
- 대응: 시작 시 `yt-dlp`, `ffmpeg` 존재 여부를 먼저 검사한다.

### Discord voice 처리 복잡도
- 대응: 보이스 계층을 분리해 테스트 가능한 구조로 만든다.

## 무엇부터 시작할까
가장 먼저 할 일은 기능 구현이 아니라 **프로젝트 뼈대와 실행 확인**이다. 아래 순서로 시작하는 것이 좋다.

1. Go 모듈 초기화
2. `cmd/bot/main.go` 생성
3. `internal/config`에서 환경 변수 읽기
4. `discordgo`로 로그인 가능한 최소 봇 실행
5. 특정 길드에 테스트용 슬래시 커맨드 1개만 등록

첫 번째 목표는 다음 한 문장으로 정리된다.

> "봇이 켜지고, Discord에 접속하고, 슬래시 커맨드 하나가 보인다."

이 목표가 되면 그 다음부터 `join`, `play`, `queue` 순서로 확장하면 된다.

## 권장 첫 작업 순서
### 작업 1
- `go.mod` 생성
- `cmd/bot/main.go` 작성
- 환경 변수 `DISCORD_TOKEN` 읽기

### 작업 2
- Discord 세션 생성
- 연결 성공 로그 출력
- 종료 시 세션 닫기

### 작업 3
- `/ping` 또는 `/join` 중 하나만 먼저 등록
- interaction 응답이 실제로 오는지 확인

이 세 단계가 끝나면 본격적으로 플레이어 구조를 붙이는 것이 맞다.

## 구현 원칙
- 모든 블로킹 작업은 `context.Context`를 받는다.
- 외부 프로세스 실패는 감추지 않고 로그로 남긴다.
- 길드별 상태는 공유하지 않는다.
- 고루틴은 시작보다 종료 전략을 먼저 설계한다.
- 초기에는 기능보다 동작 안정성을 우선한다.
